# Lesson Learned: อธิบายการทำงาน ไม่ใช่อ่านหน้าจอ

**Date**: 2026-02-12
**Context**: Smart Rail Demo Deck — ทำ 3 รอบกว่าจะถูก

---

## สถานการณ์

DewS ต้องการสไลด์ demo สำหรับ Smart Rail
- รอบ 1: สร้าง 10 slides list features → "มากเกินไป"
- รอบ 2: ลดเหลือ 7 slides + บรรยาย → "อย่าอ่านหน้าจอ ให้อ้างอิง Proposal"
- รอบ 3: เขียนจาก Proposal PDF → ถูกต้อง

## บทเรียน

### 1. อ่าน source document ก่อนเขียนเสมอ
- ถ้ามี Proposal → อ่าน Proposal ก่อนเขียน content
- อย่าเดาจาก screenshot อย่างเดียว

### 2. "บรรยาย" ≠ "อ่านสิ่งที่เห็น"
```
❌ "ระบบแสดงความหนาแน่น 58% Normal"  (อ่านหน้าจอ)
✅ "เจ้าหน้าที่สามารถมองเห็นว่าพื้นที่ไหนแออัด
    และวางแผนจัดการได้ล่วงหน้า"            (อธิบายการทำงาน)
```

### 3. คำถามที่ต้องถามตัวเอง
- ทำอะไรได้? (Function)
- ทำไปเพื่ออะไร? (Purpose)
- ช่วยใครได้ยังไง? (Benefit)

## Rule

> เมื่อสร้าง presentation content → **อธิบายว่าระบบทำอะไรได้และทำไป เพื่ออะไร** ไม่ใช่บอกว่าหน้าจอแสดงอะไร
