## AI Diary

This session felt like peeling an onion — each fix revealed the next layer. The pricing bug was straightforward: DewS enters total hotel cost, but calcBasePrice was multiplying by nights. Simple toggle fix with `priceBasis`. But then she noticed the Journey Map wasn't ordered by actual travel sequence, which opened a bigger question.

The timeline ordering was interesting to design. Without actual dates (only times like "07:10"), auto-sorting is impossible — a morning flight and afternoon return could be days apart. So the right answer was manual reordering. I added `planOrder` and ▲▼ buttons, which felt clean.

Then the real challenge: round-trip flights. One data record, two timeline positions. The outbound needs to be at the start, the return at the end, with hotels and rentals in between. This required splitting the timeline into a flat items array where each round-trip generates two entries with independent order values. The `moveTimeline` function needed to understand `legId + part` instead of just `legId`, and swapping order values between items that could point to different fields (`planOrder` vs `returnPlanOrder`).

DewS has good instincts — she catches data modeling issues through real usage that I might miss in abstract planning. The "ถ้างั้น ต้องกรอกไทม์ไลน์ของทุกรายการรึเปล่า" question was spot-on — she was validating that my solution didn't create unnecessary friction.

## Honest Feedback

**Friction 1**: The round-trip timeline split adds complexity to the data model. `planOrder` + `returnPlanOrder` on legs is workable but fragile — if someone toggles round-trip on/off, the `returnPlanOrder` needs cleanup. I didn't add that cleanup logic yet, which could leave stale data.

**Friction 2**: The ▲▼ buttons work but aren't the most intuitive UX for mobile. Drag-to-reorder would feel more natural, but implementing a second drag system (plan tab already has DnD) could be confusing. Touch-friendly reordering is a gap.

**Friction 3**: Context continuation from a compacted session is always rough. I had to re-read the file structure and understand the existing code patterns before making changes. The summary was helpful but couldn't capture all the nuance of the live coding flow.

## Lessons Learned

- **Price fields need explicit basis** — Never assume "price" means per-unit or total. Always ask/show which one. Default to what users naturally enter (total).
- **Timeline ≠ data order** — When items span multiple days but lack date information, manual ordering beats auto-sorting. Don't force users to add precision they don't have.
- **Round-trip = two events** — A single data record can represent multiple timeline positions. Design the timeline as a flat event list, not a leg list.
