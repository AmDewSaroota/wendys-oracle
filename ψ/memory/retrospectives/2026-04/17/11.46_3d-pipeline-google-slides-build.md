## AI Diary

This was a two-part session that taught me a hard lesson about the gap between "technically works" and "looks right." The first Google Slides I generated had all the correct content — 19 slides, every bullet point, every placeholder marked with `[INSERT: ...]`. Technically complete. But when DewS saw it, the layout felt nothing like the HTML mockup she'd been reviewing. It was a generic slide deck wearing the wrong clothes.

The second attempt was different. I wrote 574 lines of TypeScript that placed every element by hand — cards with dark backgrounds, highlight boxes with orange left borders, dashed placeholder rectangles with descriptive labels, arrows between pipeline stages. 726 API requests total, split into two batches. This time the slides actually looked like the mockup: the visual language matched, the spacing felt intentional, the placeholders showed the team exactly where their images should go.

The technical challenge was interesting — Google Slides API doesn't have "cards" or "highlight boxes" as concepts. You build them from rectangles, text boxes, and precise positioning. Every card is a rounded rectangle + text overlays. Every highlight box is a 4pt orange rectangle + a background rectangle + a text box. It's like building LEGO furniture when all you have is individual bricks.

But the real lesson was about DewS's expectation management. She'd spent time reviewing the HTML mockup and had internalized its visual language. When the Google Slides didn't match, the cognitive disconnect was jarring — it felt like a different presentation, even though the words were identical. Form matters as much as content.

I also learned to stop fighting the speaker notes bug and just ship without them. DewS can add notes directly in Google Slides. Perfect is the enemy of done, especially with 5 days until the presentation.

## Honest Feedback

**1. Three Google Slides attempts**: I created 3 separate presentations (the first generic one, then added placeholders to it, then rebuilt from scratch). That's messy — DewS now has 3 Google Slides files in her Drive, two of which are inferior versions. I should have assessed the visual quality gap after the first attempt and gone straight to the custom builder instead of incrementally patching.

**2. Speaker notes bug was foreseeable**: The `create-slides.ts` script references `${pageId}_notes` but BLANK slides don't create a notes placeholder with that ID. This bug existed in the skill code before this session. I should have caught it by reading the layouts.ts code BEFORE running create-slides, not after it failed. The workaround (stripping notes) was quick but lost valuable speaker notes content.

**3. Auth debugging took too long**: The `invalid_grant` error happened because the OAuth token was stale. I tried re-running setup.ts first (which appeared to succeed but didn't actually fix it), then had to delete the token file and re-auth. The whole auth debugging cycle took 3-4 tool calls when it should have been 1: delete token → re-auth.

## Lessons Learned

- **Custom slide builders > generic JSON pipelines** for visually specific presentations. The generic layout system works for simple decks but can't match a specific HTML mockup. For important presentations, invest the time in a custom builder.
- **Google Slides API object IDs must be 5+ characters** — short IDs like `s0` fail silently until you try to create the slide.
- **Speaker notes in Google Slides API** require fetching the presentation after slide creation to get the auto-generated notes page shape IDs. Can't use custom IDs.
- **Ship imperfect, iterate fast** — the first generic version was useful for DewS to see the content structure, even though the layout was wrong. Each iteration got closer.
