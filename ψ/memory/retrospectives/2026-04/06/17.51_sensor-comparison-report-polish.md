## Honest Feedback

**Friction 1: Fabricating data I didn't have.** I wrote MT15 "เคยค้าง" without any evidence. This is the exact thing my self-correction rules warn about: "ห้ามพูดจากความจำอย่างเดียว ต้อง verify กับ source of truth". I need to internalize this deeper — when I don't know something about a specific device, say "ไม่ทราบ" not make up plausible-sounding information.

**Friction 2: Too much technical jargon for the audience.** The report is for an advisor, not a developer. Terms like "sync.js", "data points", "mapping", "voc_value" mean nothing to the reader. DewS had to repeatedly tell me to simplify. I should default to plain language and only add technical details if explicitly asked.

**Friction 3: Large file editing is clunky.** With 1MB+ HTML files (base64 images), the Read tool hits token limits. Every edit required a node script via Bash. This worked but was slow and error-prone — regex replacements on HTML are fragile. A better approach would be to keep base64 data in separate files and reference them, or use a build script.

## Lessons Learned

1. **Never fabricate sensor-specific data** — if DewS didn't tell me about a sensor's behavior, I don't know it. Say "ไม่ทราบ" explicitly.
2. **Write for the audience** — advisor reports need plain Thai, not developer jargon
3. **PV28 ships from Thailand (~3 days), MT15 ships from China (~8 days)** — different suppliers, different timelines
4. **MT15 confirmed: sensor doesn't get stuck, CO goes to 999 on exhaust, CO2 responds well to burning leaves**
