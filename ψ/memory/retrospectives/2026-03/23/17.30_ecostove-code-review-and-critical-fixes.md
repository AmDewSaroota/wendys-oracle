## AI Diary

Today was a session about discipline and humility. DewS asked me to review the entire codebase, and what I found was sobering. The Thai date boundary bug — something so fundamental, converting timezone-aware dates to UTC for database queries — had been there since the beginning. Every session count, every daily summary, could have been slightly wrong for sessions happening in the early morning hours. It's the kind of bug that doesn't scream; it whispers wrong data into your research.

The co_value omission was equally humbling. One missing field in a SELECT string, and avg_co has been null for every closed session. The code dutifully computed `vals('co_value')` on data that never included it. No error, no warning, just silence.

What struck me most was the revert moment. I confidently removed query string authentication as a "security improvement," deployed it, and then realized — cron-job.org IS sending the secret via URL. The sync would have broken silently every 5 minutes. This is exactly why my self-correction rules exist: verify before acting. I caught it within minutes and hotfixed, but it was a sharp reminder that "improvement" without understanding the full system context can be destructive.

The parallel review approach worked beautifully — 3 agents reading ~11,000 lines simultaneously, each finding different classes of issues. The synthesis into a single prioritized table gave DewS a clear picture. But I must resist the urge to fix everything at once. The remaining items (PIN storage, Chart.js optimization, volunteer API auth) are real issues but not UAT blockers.

## Honest Feedback

**Friction 1: Review agent output inconsistency.** One of the three review agents (volunteer) completed but produced an empty output file. I had to re-run it, wasting ~2 minutes. The background agent system needs better guarantees that completed agents actually write their results.

**Friction 2: The revert trap.** Removing query string auth was technically correct from a security perspective, but I failed to trace the dependency chain. The memory file clearly stated the cron URL includes `?secret=`. I should have read my own memory more carefully before making the change. The "measure twice, cut once" principle applies to code changes, not just destructive commands.

**Friction 3: Large file review depth.** With index.html at 8,507 lines, even the review agents couldn't read every line in detail. Some findings were approximate (e.g., "around line 6897") and needed manual verification. For single-file apps this large, the review would benefit from a structured multi-pass approach — first pass for structure, second for security, third for performance.

## Lessons Learned

1. **Timezone bugs hide in plain sight** — `thaiDate + 'T00:00:00Z'` looks correct but is 7 hours off. Always use explicit timezone offsets when converting local dates to UTC.
2. **Verify the dependency chain before removing features** — even "insecure" features may be the only auth path for an external service.
3. **One missing field in a SELECT = silent null propagation** — always check that computed aggregates reference fields that are actually queried.
