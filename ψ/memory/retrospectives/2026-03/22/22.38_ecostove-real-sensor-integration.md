## AI Diary

This session felt like the moment a prototype becomes real. For weeks we've been building with mock data — fake house names, fake sensor IDs, fake readings. Today DewS physically held 12 sensors, paired them one by one with SmartLife, and tried to run the actual volunteer flow on her husband's phone. And naturally, everything that could go wrong, did.

The mock-to-real transition exposed layers of assumptions. The batch Tuya API rejects the entire request if even one device ID is invalid — so 9 mock IDs poisoned the one real ID. The sync schedule blocked weekends. The volunteer page doesn't talk to Tuya directly — it looks for sessions that only sync.js can create. Each discovery cascaded into the next.

What struck me was DewS's patience. She's testing on a Sunday evening, borrowing her husband's phone to simulate a volunteer, and when the sensor showed as "MT13W 7" instead of her sticker label "ES-05", she didn't panic — just said "ชื่อน่าจะสลับ" and we moved on. That pragmatism is what makes this project work.

The SmartLife name mismatch is still unresolved for 11 of 12 sensors. We'll need to map each one by turning them on individually. But at least now one sensor works end-to-end: Tuya → sync.js → session → volunteer page → baseline → cooking → manual TVOC/CO → DB. That's the full pipeline proven.

## Honest Feedback

**1. Mock data → real data transition was painful.** We had no migration script, no checklist, no way to verify the mapping. I ended up writing ad-hoc Node scripts to query Supabase and Tuya APIs. This should have been planned better — a simple admin tool to "import sensors from Tuya Cloud" would have saved 30 minutes of debugging.

**2. I forgot about the rest day logic.** When sync returned "rest day (วันอาทิตย์)", I had to trace through the code to find the active_days config. This is a setting that's invisible until it blocks you. The admin dashboard should show the current schedule status prominently — "Today: rest day" or "Today: active, next sync in 3 min".

**3. The sensor ID mapping problem is a UX gap.** DewS physically labeled sensors with stickers (ES-01 to ES-12), but SmartLife assigned arbitrary names during pairing. There's no way to match them without turning on one at a time. A better approach: the admin page could show "Device X just came online — which sticker is on it?" to build the mapping interactively.

## Lessons Learned

- Batch APIs can fail entirely if one item is invalid — always separate real from mock data
- LINE OA cron should respect the same schedule as data sync
- Pagination with "load more" is the simplest performance fix for large tables
