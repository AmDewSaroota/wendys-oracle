## AI Diary

This was a session of finishing what was started and then pivoting to a UX insight that DewS caught in real-time. The modal work was straightforward — translating flowmap HTML into guide-prefixed equivalents for Flows 4, 5, and 6. Flow 4's house modal was the most complex with cascading province/district/subdistrict dropdowns and conditional sensor fields. But the real moment came when DewS sent screenshots of the overview table and asked a simple question: "ควรเปลี่ยนตรงไหนดี?"

She was right. Having editable dropdowns in both the overview AND the management tabs creates confusion about where the "source of truth" is. The overview should be a dashboard — read, observe, understand — not a place to make changes. Same for the subjects table's sensor dropdown: if you want to change sensor binding, click "แก้ไข" and do it properly through the edit modal.

This is the kind of UX thinking that comes from someone who actually imagines volunteers using the system. Not abstract "best practices" but concrete "จะสับสนไหม?" I respect that DewS catches these things at midnight while getting sleepy. She still sees the friction before it reaches real users.

The session also revealed a pattern: DewS gives instructions in batches — "รับคำสั่งทั้งหมดก่อน เสร็จแล้วค่อยแก้นะ" — collect all requirements, then execute. This is efficient and prevents context-switching mid-implementation. She gave 4 items for tomorrow, then said "ลัดมาแก้อันนี้ก่อน" for the dropdown fix. Priority override, clear and decisive. I like working this way.

## Honest Feedback

**Friction 1: Context restoration overhead.** This session started from a context summary, meaning I had to re-read the exact same files I'd read in the previous session to find insertion points. The summary told me "insert after line 1628" but line numbers shift after edits, so I had to re-search every time. This is unavoidable but adds ~10 minutes of purely mechanical work at the start of every continuation.

**Friction 2: Duplicate dropdown logic scattered.** The overview table, subjects table, and potentially other places all had independently-written dropdown rendering code. When DewS said "deactivate dropdowns," I had to hunt for each instance separately. If the rendering logic was centralized (render function per column type), one change would fix all. But in a single-file 9000+ line HTML app, this kind of refactoring is risky mid-project.

**Friction 3: No visual verification.** After deploying the dropdown→readonly change, I can't actually verify it looks right. I'm trusting that the HTML I wrote renders correctly. DewS will see it tomorrow and might say "the text is misaligned" or "the color doesn't match." A browser preview or screenshot capability would catch these before deploy.

## Lessons Learned

- **"Where should changes be made?" is a UX design question, not a technical one.** Dropdowns everywhere = convenience but also confusion. Read-only overview + explicit edit buttons = clear mental model for users.
- **Batch instructions before executing** — DewS's pattern of "listen to all requirements first, then work" prevents mid-task context switches and produces cleaner commits.
