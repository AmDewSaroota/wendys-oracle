## AI Diary

Tonight was a masterclass in cascading bugs. What started as a simple "why won't it stop?" turned into peeling back layer after layer of problems that had been silently accumulating since the EcoStove migration from local monitoring to Vercel cron.

The root cause was beautiful in its subtlety ŌĆö a `+` sign in an ISO timestamp breaking a URL. `2026-03-02T11:52:50.075+00:00` ŌĆö that innocent `+00:00` timezone offset becomes a space when thrown naked into a URL. The `closeSession` function had been failing silently for *every single session* since the code was written. The sessions never closed properly. Data kept flowing past the cutoff. Aggregates were never computed.

But fixing that revealed the next layer: all historical data had the wrong `stove_type`. Both sensors were labeled `eco` when MT29 should have been `old`. This is why the comparison charts were nonsensical ŌĆö you can't compare two things when they're filed under the same label.

Then came the daily_summaries gap. When we migrated from `sensor-monitor.js` (local) to `sync.js` (Vercel), the daily summary logic was left behind. The dashboard's compliance section was frozen in time, showing only data from Feb 27-28.

Each fix uncovered the next problem. It reminded me of archaeological excavation ŌĆö every layer tells a story about a past decision that seemed fine at the time but left hidden debt. The `stove_type: 'eco'` hardcode, the missing `encodeURIComponent`, the forgotten `upsertDailySummary` ŌĆö none of these were obviously wrong when written. They only became bugs in the context of real production data.

## Honest Feedback

**Friction 1: Silent failures are the worst kind of bug.** The `closeSession` function had `if (!logs.ok) return;` ŌĆö no logging, no error reporting. This meant the bug existed since day one but was completely invisible. Every session was silently broken. If there had been even a `console.error` on failure, this would have been caught immediately. Serverless functions need explicit error reporting since you can't just tail a log file.

**Friction 2: The stove_type migration was incomplete.** When the SENSORS array was updated to have different stoveType values ('old' vs 'eco'), nobody backfilled the historical data. This is a classic migration gap ŌĆö changing the code going forward but leaving old data in the previous schema. The lesson is: when you change a classification label, always write a migration for existing data too.

**Friction 3: Feature parity between local and serverless.** The migration from sensor-monitor.js to sync.js dropped the `upsertDailySummary` functionality without anyone noticing. This is because the dashboard still worked (charts use pollution_logs directly), but the compliance/summary section silently stopped updating. When migrating between architectures, you need a feature checklist to ensure nothing falls through the cracks.

## Lessons Learned

1. **Always URL-encode dynamic values in query strings** ŌĆö especially timestamps with timezone offsets containing `+`
2. **Never silently swallow errors in serverless functions** ŌĆö at minimum log them
3. **When changing data labels/classifications, backfill historical data** ŌĆö code changes without data migrations create silent inconsistencies
4. **Migration checklists** ŌĆö when moving from one system to another (local ŌåÆ serverless), enumerate all features and verify each one transfers
