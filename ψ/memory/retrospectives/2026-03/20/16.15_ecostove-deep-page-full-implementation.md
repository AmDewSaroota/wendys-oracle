## AI Diary

This session was a study in surgical precision. I picked up from where the previous session crashed out of context — right at `updateResearchStats()` line 3454 — and had to reconstruct the full picture of what needed to change from the session summary alone. That's always a disorienting moment, like waking up mid-surgery and having to remember which organ you were operating on.

The key insight that made everything clean was the `renderStat()` helper function. The old `updateResearchStats()` had three separate code paths — primary fields with comparison, secondary fields without, and TVOC with its own special handling. By unifying everything into one helper, the function went from 100+ lines to about 50, and adding CO support was just one extra line.

The stacked bar was satisfying to write. It's a purely HTML-rendered visualization — no Chart.js needed — and the math is elegant: ambient = gross - net. The three scenario variants (normal with green "saved" section, no-eco with placeholder, eco-fail with red bars) each tell a different story using the same underlying structure.

The biggest architectural decision was separating the drilldown system from the main charts. In the old code, drilldown would overwrite the same canvases. Now, main charts always show daily avg, and drilldown creates its own canvases in a separate panel. This is cleaner, less stateful, and means the user never loses the overview when drilling down.

I was nervous about the `co_value` and `tvoc_value` addition to `AGG_FIELDS`. These are manual data fields, and I worried they might cause issues with the daily aggregation since they'd have many null values. But the aggregation logic already handles nulls gracefully — it only increments count when the value isn't null. So days with no CO/TVOC readings simply won't have entries in those fields, and the charts will show gaps naturally.

## Honest Feedback

**Friction 1: Context continuation is lossy.** The session summary captured the *what* perfectly but lost the *feel* — the mental model of how all the functions interrelate. I had to re-read 500+ lines of code to rebuild that understanding. The plan file helped, but the plan was written for the original implementation, not for the mockup-to-production adaptation. Next time, a brief "current mental model" section in the handoff would save 20 minutes.

**Friction 2: Single-file architecture is hitting its limits.** `index.html` is now 7000+ lines. Finding the right insertion points, making sure edit strings are unique, and verifying that removing a function doesn't break something 3000 lines away — all of this is getting harder. But DewS chose this architecture for deployment simplicity (single file on Vercel), and that trade-off still makes sense for this project's scale.

**Friction 3: No automated testing.** I verified correctness by grepping for broken references and checking function signatures, but there's no way to actually test that the charts render correctly without opening a browser. The deploy-and-check cycle is the only real test. For a research dashboard that's acceptable, but if the project grows, a basic Playwright smoke test would catch the obvious breaks.

## Lessons Learned

- **Unified helpers beat copy-paste variants.** `renderStat()` replaced 3 separate rendering paths and made adding new pollutants trivial.
- **Separate drilldown from overview.** Dynamic canvas creation in a container is cleaner than reusing shared canvases with conditional rendering.
- **Manual data fields work fine in daily aggregation** as long as the aggregation logic handles nulls (which `count` tracking naturally does).
