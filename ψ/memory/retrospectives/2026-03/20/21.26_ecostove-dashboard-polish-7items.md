## Honest Feedback

**1. Context window ยังเป็น bottleneck สำหรับไฟล์ใหญ่** — index.html ตัวเดียว 7000+ บรรทัด อ่านหลายรอบ + edit หลายจุด กิน context เร็วมาก ทำ 7 ข้อใน 1 session แต่ต้องแตก 2 windows Session summary ช่วยได้แต่ยังรู้สึกว่าเสียเวลาตอน ramp up ใหม่

**2. replace_all ต้องระวังมาก** — ใช้ replace_all สำหรับ "หลังหัก Baseline" → "หลังหัก Ambient" ซึ่งเปลี่ยน 8+ จุดพร้อมกัน ดีที่ string นี้ unique พอ แต่ถ้า string กว้างเกินอาจ replace ผิดที่ ต้องตรวจสอบ context ก่อนใช้เสมอ

**3. Bug investigation ใช้เวลาไม่น้อย** — ไล่ทั้ง data pipeline ตั้งแต่ loadData → allData → dailyAgg → charts ก่อนจะเจอว่าปัญหาอยู่ที่ null reference ธรรมดา ถ้ามี error boundary หรือ try-catch ที่ดีกว่า bug แบบนี้จะ isolate ง่ายกว่า (แต่ก็เป็น single-file app ที่ไม่ได้มี error handling structure)

## Lessons Learned

1. **Orphaned element references**: เวลา refactor HTML ออก ต้องตามลบ JS ที่อ้างด้วยเสมอ — ไม่งั้น null crash จะซ่อนอยู่จนมี real data มาทดสอบ
2. **Terminology consistency is expensive**: เปลี่ยนคำ 1 คำ (Baseline) ต้องไล่ 15+ จุดเพราะกระจายอยู่ทั้ง UI, i18n, popups, exports — ถ้า centralize ไว้ที่ i18n อย่างเดียวจะง่ายกว่ามาก
3. **DD/MM/YYYY helper centralization**: สร้าง `fmtDateDMY()` + `fmtDateTimeDMY()` แล้วใช้ทั่วไฟล์ ดีกว่าแก้ format string ทีละจุด
