I wasted plan-time designing creation steps for files that already existed. Need to verify agent claims against actual file system before building plans around them.

The summary was good but I had to re-read files to understand the exact state. The `/forward` skill helps bridge sessions, but within-session compaction still loses nuance. **Friction 3 — Bash tool had intermittent failures at session start.** Multiple `echo "hello"` commands returned exit code 1. Eventually it started working. ## Lessons Learned

1. **Always read the code before estimating API usage.** My initial quota analysis was wrong because I assumed per-device calls. The batch API design changes the math completely. Don't guess architecture — read the implementation.
2. **Verify subagent claims against reality.** Explore agents can report incorrect file existence. A quick `Glob` check is faster and more reliable than trusting agent output for basic file existence.
3. **Dynamic over hardcoded.** The LINE OA cron had hardcoded device arrays. This works for 2 devices but breaks at 10. Always prefer database-driven configuration when the data is already in the DB.
