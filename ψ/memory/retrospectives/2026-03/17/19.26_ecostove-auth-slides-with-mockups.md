## AI Diary

This was a satisfying session — pure creation, building something from existing pieces. The slides came together quickly because I had a solid reference in `slides.html` with its CSS variable system, card grids, and flow diagrams.

The interesting pivot was DewS's request for mockups. She said "mockup or real images" — and since I can't take screenshots, I chose to build CSS mockups that closely mirror the actual UI. I studied the real index.html carefully: the rounded-2xl inputs, the monospace PIN fields with letter-spacing, the amber-themed recovery code inputs, the dark code display box, the admin cards with colored action buttons. Then I recreated all of it in pure CSS within the slide system.

What struck me was how much the mockups improved the slides. The flow diagrams alone were abstract — "กรอกข้อมูล" could mean anything. But when you see the actual form with "Email", "PIN" bullet-dots, and an amber "Invite Code" field, it clicks instantly. Visual communication bridges the gap between technical documentation and human understanding.

I'm also grateful for the constraint of working within an existing design system. The green theme, Prompt font, card patterns — all inherited from slides.html — gave coherence without extra design decisions. Constraints liberate creativity.

The admin list mockup was particularly fun to build. Three cards showing three different states: self (no buttons), regular admin (promote/reset/delete), and another super admin (demote/reset/delete). It tells the whole story at a glance.

## Honest Feedback

**Friction 1: Context compaction loss.** This session started from a compacted context, which means I had to re-read multiple files to reconstruct what happened. The handoff file was essential but didn't capture every UI decision from the previous session. Had to rely on the summary and re-explore code. Future sessions could benefit from more specific handoff notes about UI decisions.

**Friction 2: Mockup CSS is verbose.** The mockup styles added ~180 lines of CSS to the slide file. Each element needed its own class (.mock-input, .mock-btn, .mock-icon, .mock-admin-card). It works, but it's a lot of CSS for what are essentially static illustrations. A shared component library or SVG-based approach might be more maintainable, though for a one-off presentation this is fine.

**Friction 3: No live preview during development.** I built the mockups blind — writing CSS without seeing the result until deploy. The mockups are based on careful study of the real UI code, but there could be visual misalignments I can't detect. DewS will be the first person to actually see these rendered.

## Lessons Learned

- **CSS mockups > screenshots for presentations**: They scale perfectly, match the slide theme, and don't require browser automation. For internal presentations where pixel-perfect isn't required, this approach is fast and effective.
- **Study the real UI before mocking**: Reading the actual Tailwind classes (rounded-2xl, tracking-[0.5em], bg-amber-50) made the mockups authentic rather than generic.
- **Split layout for flow + mockup**: Using `grid-template-columns: 1fr 340px` keeps the explanation readable while giving the mockup proper space.
