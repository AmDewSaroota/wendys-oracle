## Honest Feedback

**Friction 1: Context window pressure.** This session hit context limits and had to be summarized and continued. The sheer volume of changes across multiple files (index.html alone is 8000+ lines) made it hard to keep everything in working memory. I had to re-read sections multiple times after compaction. For future sessions, it would be better to batch changes by file rather than by feature.

**Friction 2: CSS class naming collision risk.** When adding mock/flow CSS to index.html, I had to prefix everything with `guide-` (e.g., `guide-mock-nav` instead of `mock-nav`) to avoid conflicts. This works but creates a parallel naming world. If the flowmap page ever updates its styling, the dashboard guide won't automatically follow. This is a maintenance debt.

**Friction 3: Wording drift is systemic.** The fact that ~40 mismatches existed between guides and the real UI suggests a deeper problem: there's no single source of truth for UI labels. The i18n dictionary defines what the app shows, but guides are manually written separately. Any UI text change requires updating 3 places (i18n dict, flowmap guide, built-in guide). This will drift again.

## Lessons Learned

- **Simulate before shipping**: DewS loved seeing 6 LINE message scenarios before committing to Format C. Simulation builds confidence.
- **Split large sections**: The combined "อนุมัติ + แอดมิน" section was too dense. Splitting it made each section focused and scannable.
- **Prefix CSS when embedding**: When bringing CSS from one page into another, always namespace to prevent collisions.
