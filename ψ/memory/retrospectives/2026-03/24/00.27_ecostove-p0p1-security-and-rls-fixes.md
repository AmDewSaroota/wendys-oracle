## Honest Feedback

**Friction 1: RLS lockdown was too aggressive.** I should have thought through the architecture more carefully before writing migration 023. The dashboard uses anon key for ALL writes, not just reads. This caused a production break that DewS had to debug live ("ทำไมลงทะเบียนเพิ่มไม่ได้ล่ะ"). The fix (migration 024) was quick, but the mistake was avoidable. Always trace the full write path before restricting permissions.

**Friction 2: Context window ran out before /rrr completed.** The session was productive but long (~4.5 hours). By the time DewS asked for the retrospective, context was nearly exhausted. The conversation had to be continued in a new session just to write this document. Should prioritize creating checkpoints mid-session for long coding sessions.

**Friction 3: Sensor naming drift not caught earlier.** DewS changed naming from ES-XX to BS-XXX at some point, and my memory still had the old convention. I should have flagged the memory update proactively when I first saw "BS 001" in the database, rather than waiting for DewS to mention it.

## Lessons Learned

1. **RLS must match actual architecture, not ideal architecture.** If your frontend uses anon key for writes, locking anon to SELECT breaks everything. Trace the full CRUD path before restricting.
2. **Physical-digital mapping is permanent when hardware IDs are involved.** Tuya Device IDs are burned in — sticker labels are safe to apply before data wipes.
3. **Migration chains can conflict.** 023 locked down, 024 re-opened. The net effect is correct (admin_users locked, everything else open for admin ops), but the journey left a production break in between.
4. **Hero card comparison needs both sides of the comparison.** The eco-vs-old design requires actual old stove data. Until then, `--` is correct behavior.
