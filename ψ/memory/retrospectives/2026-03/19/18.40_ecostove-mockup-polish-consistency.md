## Honest Feedback

**Friction 1: Consistency blindness.** I added 3 different hero sections at different times and never checked that their footer formats matched. This is a recurring pattern — when building incrementally, I lose sight of cross-component consistency. I should do a "consistency audit" after any multi-section change: same data format, same units, same counts everywhere.

**Friction 2: Sample data shortcuts.** When I first built the mockup tables with 4 rows, I was thinking "representative sample" but DewS was thinking "accurate representation." A mockup for a researcher is NOT a wireframe for a designer — every number is a claim. I should default to showing ALL items (10 houses = 10 rows) and only reduce if DewS explicitly says to summarize.

**Friction 3: Multiple deploys.** We deployed 4 times this session. Each deploy was triggered by a catch that should have been caught in the first pass. If I had done a "professor audit" before the first deploy (checking all numbers, all formats, all counts), we could have shipped once. The deploy-fix-deploy cycle wastes time and creates unnecessary Vercel build history.

## Lessons Learned

- **Mockup ≠ wireframe when the audience is a researcher** — every number must be accurate, every count must match the project spec (10 houses), every format must be consistent across scenarios
- **Cross-section consistency audit** — after building multi-scenario content, always check that shared elements (footers, counts, labels, formats) match across all variants
- **"What's the rule?" test** — before finalizing any data display, ask: what's the logic that determines this content? If there's no clear rule, the display is arbitrary and will be questioned
