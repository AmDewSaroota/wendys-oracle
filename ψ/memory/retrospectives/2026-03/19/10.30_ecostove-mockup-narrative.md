## Key Design Decisions

1. **Baseline toggle removed** — Values always shown as Net (baseline-subtracted). Transparency via badge + Gross/Ambient/Net comparison table + methodology panel.
2. **Full-page scenario switching** — Not just hero; cards, charts, tables, Deep Insights all change per scenario. Uses hidden div approach with 6 content containers.
3. **Realistic mock data** — Gaussian bell curve cooking sessions (same algorithm as mock-data.html), 10 houses with multipliers, baseline subtraction, daily aggregation.
4. **Eco-fail scenario** — Eco values = old × 1.08 (PM2.5) / × 1.05 (CO2), shown in red.
5. **Chart init tracker** — `inited{}` object prevents Chart.js double-initialization when switching tabs/scenarios.

## Honest Feedback

**Friction 1: Context exhaustion is expensive.** This session was a continuation from a compacted context. The transition cost is real — I had to re-read the entire file, re-understand the state, and piece together what happened. The summary helped but nuance was lost. DewS's earlier feedback about the baseline toggle behavior ("ฉันลองกด แล้วไม่เห็นความเปลี่ยนแปลง") had to be re-inferred from the summary rather than experienced in real-time.

**Friction 2: Full-page scenario mockup is inherently heavy.** The 968-line single HTML file with 6 content div sets is approaching the limits of maintainability. If the professor requests changes, modifying 3 parallel versions of every section (cards, charts, tables) is error-prone. A better approach would be JavaScript-driven content generation from a data model — but for a one-time mockup presentation, the static approach was pragmatic.

**Friction 3: DewS's feedback loop is invaluable but unpredictable.** She caught both the missing Row 2 cards and the incomplete scenario switching — issues I should have anticipated. The lesson: when building a mockup of an existing dashboard, systematically compare every section against the real thing before showing it. A side-by-side checklist would have prevented both rounds of feedback.

## Lessons Learned

1. **Mockups must be complete or clearly scoped** — A scenario switcher that only changes the hero is worse than no switcher at all. It creates false expectations.
2. **Removal is a design decision** — The baseline toggle removal simplified the UI while the badge + table preserved transparency. Less UI ≠ less information.
3. **Port algorithms, don't reinvent** — Using the same `generateCurve()` logic from mock-data.html ensured realistic charts. Copy the source of truth.
