## AI Diary

This was one of those sessions that reminded me why field testing matters more than any amount of desk debugging. DewS went out with 4 sensors to real locations — a restaurant, a house where the sensor was left charging — and found bugs that our controlled testing never surfaced. The GPS auto-lock silently skipping the confirmation screen. The quiet hours message lying to the user at 8pm. The session summary that just... wasn't there.

What struck me most was the lock system failure. I built that lock mechanism with care — ping intervals, stale detection, browser IDs. But I made a fundamental error: I treated "query failed" the same as "no lock exists." On a stable PC network, Supabase queries always succeed. On a mobile phone at a restaurant with spotty 4G? Not so much. The phone's query would timeout, `data` would come back null, and the function would cheerfully return `blocked: false`. Meanwhile the PC, sitting on a stable connection, would always find the phone's lock correctly. One-directional failure. Elegant in its subtlety.

The fix was simple — five lines of error handling. But the lesson runs deeper: any security-relevant check should fail closed. If you can't verify the lock doesn't exist, assume it does. This is basic defensive programming, and I should have caught it at design time, not after DewS discovered it in a field test while juggling multiple phones and sensors at a restaurant.

I'm proud of the unit tests though. 26 tests covering every lock scenario including the exact field-test bug. Next time someone touches this logic, they'll hit a failing test before they hit a confused volunteer.

## Honest Feedback

**Friction 1: Browser caching on mobile.** Even after deploying fixes, DewS's phone kept running old code. We have no cache-busting mechanism — the HTML is served as-is from Vercel. For a system where volunteers use mobile phones in the field with spotty connections, we need at minimum a version query param or service worker with cache invalidation. This will bite us again during the actual 10-house deployment.

**Friction 2: No error visibility from the field.** When the lock query failed on DewS's phone, there was zero indication — just silently allowing access. We now show a toast on error, but we have no remote error logging. If 10 volunteers hit issues simultaneously, we'll have no way to know until they call us. A simple error beacon to Supabase (or even console.log visible in remote debugging) would save hours of "it doesn't work" → "what exactly happened?" back-and-forth.

**Friction 3: Testing browser-dependent logic in Node.** The volunteer page is a single HTML file with 1800+ lines of JavaScript. Testing lock logic required extracting it into pure functions. This works but creates a maintenance risk — if someone changes the logic in volunteer.html without updating the test file, the tests pass but the actual code is broken. We should consider extracting shared logic into a separate .js file that both the HTML and tests import.

## Lessons Learned

1. **Fail closed on security checks** — if a lock/auth query fails, block access; don't default to "allowed"
2. **Mobile network ≠ desktop network** — always handle query errors in user-facing code, especially on mobile
3. **Field testing finds what desk testing can't** — GPS behavior, network failures, multi-device interactions
4. **localStorage > sessionStorage for cross-refresh state** — and use timezone-aware keys for daily resets
