## AI Diary

This was a humbling session. I spent significant effort trying to be clever — saving tokens, restoring them async, adding timeouts — when the simplest solution was right there all along: just clear everything and let the user re-login. It's a 2-3 person admin team; nobody cares about session persistence across refreshes.

The Supabase SDK v2 internal session lock was a genuinely tricky problem. I didn't know going in that the SDK uses a mutex-like mechanism that blocks ALL Supabase operations (even unrelated data queries) while a token refresh is in progress. This isn't documented prominently. Each fix attempt taught me something: attempt 1 showed that the SDK lock affects all operations regardless of Promise.race wrappers; attempt 2 showed that setSession() itself triggers the same internal lock; attempt 3 showed that fire-and-forget doesn't help because the lock persists.

The shell dying mid-session was frustrating. I couldn't use dev browser to verify fixes myself, so DewS had to test manually. But it turned out fine — her direct feedback ("Login timeout อีกละ") was more valuable than automated testing would have been. Sometimes constraints force better workflows.

## Honest Feedback

**1. Supabase SDK blocking is a trap**: The biggest friction was not knowing that Supabase JS v2 uses an internal session lock. Three fix attempts failed because of this undocumented behavior. The SDK should either document this clearly or not block data queries during auth operations. I should have tried the simplest solution (clear everything) first instead of trying to be clever with token restoration.

**2. Shell instability wasted time**: The bash environment dying with no error messages (just exit code 1 on every command) meant I couldn't verify fixes myself. This forced manual testing via DewS. The root cause is unclear — possibly a Windows shell resource issue after running dev browser and multiple background tasks.

**3. Over-engineering before validating**: I wrote 3 progressively complex solutions before landing on the simplest one. The lesson is clear: when dealing with third-party SDK internals that you don't fully understand, start with the simplest possible workaround and only add complexity if needed.

## Lessons Learned

- Supabase SDK v2 `createClient()` auto-detects auth tokens in localStorage and triggers `_recoverAndRefresh()` which blocks ALL subsequent operations with an internal mutex
- When SDK internals are opaque, default to the simplest workaround (clear state) rather than trying to work around the lock
- For small admin teams (2-3 people), session persistence across refreshes is not worth the complexity cost
