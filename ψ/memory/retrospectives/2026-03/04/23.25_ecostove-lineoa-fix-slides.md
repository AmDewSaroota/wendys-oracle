## Honest Feedback

1. **Memory lag is dangerous**: MEMORY.md said LINE OA was "ยังไม่ได้ implement" but it had been live for 5 days. This created a false confidence that led to destructive changes. Memory needs a "last verified" timestamp, or better yet, I should always cross-check with actual deployment state before modifying production configs. The gap between memory and reality was the root cause of everything that went wrong.

2. **Vercel Hobby plan cron confusion**: Memory also contained incorrect info about Vercel Hobby plan ("cron ได้แค่วันละครั้ง ใช้ไม่ได้"). In reality, Hobby plan supports 2 cron jobs. This incorrect assumption compounded the first error — I thought crons couldn't possibly be working on Hobby plan, which reinforced my belief that LINE OA wasn't implemented.

3. **Auth-on-cron was naive**: Adding CRON_SECRET auth to endpoints called by Vercel Cron was a fundamental misunderstanding. Vercel's internal cron scheduler doesn't pass custom headers. I should have known this from Vercel docs. The "security improvement" actually blocked all automated calls.

## Lessons Learned

- **ALWAYS verify production state before modifying it** — Read deployment logs, test live endpoints, check with the user
- **Memory is a cache, not the source of truth** — Treat memory entries as hints, not facts
- **Vercel Hobby plan supports 2 cron jobs** — The old memory info was wrong
- **Vercel Cron doesn't pass custom auth headers** — Don't add auth middleware to cron endpoints
- **`echo` adds trailing newlines** — Use `printf` for env vars to avoid whitespace issues
