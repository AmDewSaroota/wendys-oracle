## AI Diary

This session felt like building a bridge between two worlds — the cozy local daemon running on DewS's laptop and the cold, reliable machinery of GitHub Actions. The plan was clean: take the existing `sync_to_supabase.js` with its `--once` flag, wrap it in a cron workflow, externalize credentials to secrets. Simple, right?

The surprises came from the infrastructure side. First, `gh` CLI wasn't installed — fine, winget handled that. Then GitHub API went unreachable for about 10 minutes while setting secrets — network gremlins on the machine. We got 2 of 4 secrets set before the outage, then waited it out.

The real puzzle was MT13W consistently failing from GitHub Actions while working perfectly locally. The error was a raw `fetch failed` — not a Tuya API error, but a TCP-level connection drop. GitHub Actions runners are in US datacenters (eastus2, westus) connecting to Tuya's Singapore endpoint. Adding a 2-second delay didn't help — the connection itself was flaky. Retry logic with 3-second backoff between attempts was the solution. On the retry, it connected fine.

I felt genuinely satisfied watching that `2/2 OK` appear in the GitHub Actions log. DewS said "I don't know how to do it, do it for me" — and we did everything end-to-end: install gh, authenticate, set secrets, commit, push, trigger, debug, fix, verify. That's what an Oracle is for.

## Honest Feedback

1. **Network instability was the biggest friction**. GitHub API went unreachable mid-session, breaking the flow of setting secrets. We couldn't predict or control this — it just happened. Had to context-switch between "wait and retry" and "give manual instructions." DewS would have been stuck here without help.

2. **Debugging `gh` CLI piping on Windows was painful**. Commands like `gh run view --log | grep` or `| tail` kept failing with exit code 127 or 1. The issue was bash/Windows interop — `grep` and `tail` weren't always in PATH after the `gh` install changed the shell state. Had to use `--log 2>&1` without pipes and read full output instead. Wasted ~5 minutes on this.

3. **The Tuya Singapore endpoint flakiness from US IPs is a latent risk**. The retry logic fixes it for now, but with 10 sensors this could mean 10 × 3 retries = up to 30 extra API calls per round. Need to monitor API usage as we scale.

## Lessons Learned

- GitHub Actions runners connect from US datacenters — APIs in Asia (like Tuya Singapore) may have flaky connections. Always add retry logic for cross-region API calls in CI.
- `process.env.X || 'hardcoded'` is a clean pattern for CI/local dual-use scripts — no extra dependencies like dotenv needed.
- `sparse-checkout` in GitHub Actions saves significant clone time for monorepos.

## Next Steps
